# ADR-003: Frontend Technology Stack

## Status

Accepted

## Date

2026-04-05

## Context

The app is built on a fresh Vite + React 19 + TypeScript scaffold with no existing dependencies. The PRD requires responsive layouts across desktop, tablet, and mobile; dark/light theme support with OS detection; and complex interactive UI (comparison panels, notes editor, search, cross-reference overlays). State management needs to handle Bible navigation, highlights, notes, bookmarks, reading progress, and theme preference — all persisted locally. The app needs URL-based navigation for deep linking to specific passages.

## Decision

Use **Tailwind CSS v4** for styling, **Zustand** for state management with its `persist` middleware for localStorage, and **React Router v7** in declarative SPA mode for client-side routing. No UI component library — build custom components to achieve the Apple-like aesthetic.

## Alternatives Considered

### Alternative 1: CSS Modules

- **Description**: Use Vite's built-in CSS Modules support for scoped styles per component.
- **Pros**: Zero runtime overhead; full CSS control; no build dependency.
- **Cons**: Verbose for responsive breakpoints and theme tokens; no utility-first ergonomics; harder to maintain a consistent design system.
- **Why rejected**: Tailwind provides faster development for responsive layouts and theme tokens, which are core to the PRD's responsive and dark/light theme requirements.

### Alternative 2: React Context + useReducer

- **Description**: Manage all application state with React Context and useReducer.
- **Pros**: No external dependency; built into React.
- **Cons**: Verbose boilerplate for multiple state slices; no built-in persistence; causes unnecessary re-renders without careful optimization.
- **Why rejected**: Zustand's persist middleware eliminates the need for manual localStorage integration, and its selector-based subscriptions prevent unnecessary re-renders.

### Alternative 3: No Router (State-Based Views)

- **Description**: Manage all views via state toggling instead of URL-based routing.
- **Pros**: Simpler for a single-page reading app; no extra dependency.
- **Cons**: No browser history; no deep linking; no shareable URLs for passages.
- **Why rejected**: URL-based navigation is essential for shareability and browser back/forward support, both expected by modern web users.

### Alternative 4: shadcn/ui + Tailwind

- **Description**: Use shadcn/ui component library built on Radix primitives + Tailwind.
- **Pros**: Pre-built accessible components; faster development.
- **Cons**: Components follow Material/shadcn aesthetic, not Apple-like; heavy customization needed to match the desired design; adds complexity.
- **Why rejected**: The Apple-like aesthetic requires custom component design. Building from scratch with Tailwind gives full control over the visual language.

## Consequences

### Positive

- Tailwind enables rapid responsive design with utility classes and built-in dark mode support.
- Zustand provides simple, performant state management with automatic localStorage persistence.
- React Router v7 enables shareable URLs (e.g., `/genesis/1`) and browser history navigation.
- Custom components ensure full control over the Apple-like aesthetic.
- Total added dependencies: 3 (tailwindcss, zustand, react-router).

### Negative

- Custom component library requires upfront design and CSS effort.
- No pre-built accessible primitives — accessibility must be implemented manually.
- Team must be familiar with Tailwind's utility-first approach.

### Risks

- **Accessibility without a component library**: Building accessible components from scratch requires careful ARIA markup and keyboard handling. Mitigation: reference WAI-ARIA patterns; test with screen readers early.
- **Tailwind v4 stability**: Tailwind v4 is relatively new. Mitigation: it is the current stable release with documented migration from v3.

## Implementation Notes

- Configure Tailwind v4 with custom theme tokens matching the Apple-like design system (colors, typography, spacing, shadows, border radii).
- Use Tailwind's `dark:` variant with the `class` strategy for manual theme toggling alongside `prefers-color-scheme` detection.
- Structure Zustand stores as separate slices: `bibleStore` (navigation, translation selection), `studyStore` (highlights, notes, bookmarks), `progressStore` (reading progress), `themeStore` (theme preference).
- Use React Router v7's declarative SPA mode (no SSR) with routes for: `/` (redirect to last read or Genesis 1), `/:bookId` (book overview), `/:bookId/:chapter` (chapter reading), `/search` (search results).
