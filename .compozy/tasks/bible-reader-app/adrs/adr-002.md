# ADR-002: Bible Data and Cross-Reference Strategy

## Status

Accepted

## Date

2026-04-05

## Context

The app requires Portuguese Bible text (ARA, ACF, NVI) and cross-reference data. The PRD specifies that Bible text must come from an external source with no hardcoded scripture in source code. Several options exist: external APIs (api.bible, bible-api.com), self-hosted JSON files (thiagobodruk/bible), and cross-reference datasets (scrollmapper/bible_databases). The app must perform well on mobile with constrained resources, and the PRD notes that full offline mode is out of scope but local caching is acceptable.

## Decision

Bundle Bible text and cross-reference data as JSON files imported into the application bundle. Use the `thiagobodruk/bible` GitHub repository as the source for ARA (`pt_aa`), ACF (`pt_acf`), and NVI (`pt_nvi`) translations. Use `scrollmapper/bible_databases` or `OpenBibleData` as the cross-reference data source, bundled as a separate JSON file. Bible data is transformed at build time into a normalized structure keyed by book, chapter, and verse for efficient lookup.

## Alternatives Considered

### Alternative 1: External API (api.bible)

- **Description**: Fetch Bible text on demand from the api.bible REST API.
- **Pros**: Smaller initial bundle; access to many translations; centralized data updates.
- **Cons**: Requires API key and approval; network dependency for every chapter load; rate limits; adds latency on slow connections.
- **Why rejected**: The PRD requires 2-second load times on 3G. API calls add unpredictable latency and a hard dependency on an external service's availability and rate limits.

### Alternative 2: Static JSON served from public/

- **Description**: Place JSON files in the `public/` directory and fetch them at runtime.
- **Pros**: Smaller JS bundle; data can be updated independently.
- **Cons**: Still requires network for first load; additional HTTP requests; harder to preload and cache.
- **Why rejected**: Bundling provides instant access to already-loaded data with zero network requests after the initial page load.

## Consequences

### Positive

- Zero network latency for Bible text after initial page load.
- No API key management or external service dependency.
- Predictable performance regardless of network conditions.
- Works fully in-browser with no backend.

### Negative

- Initial bundle size increases (~2-3MB compressed for 3 translations).
- Data updates require a new app deployment.
- CC BY-NC 2.0 BR license restricts commercial use of the thiagobodruk data.

### Risks

- **Bundle size**: 3 translations in JSON could be large. Mitigation: compress at build time with Vite's gzip/brotli; lazy-load per-translation data; tree-shake unused chapters if possible.
- **License restriction**: thiagobodruk/bible uses CC BY-NC 2.0 BR (non-commercial). Mitigation: document this constraint; plan migration to a commercially licensed source if needed.

## Implementation Notes

Transform raw JSON from thiagobodruk/bible into a normalized TypeScript data structure at build time or during development. The structure should map `{bookId: {chapterNumber: {verseNumber: "verse text"}}}` for O(1) verse lookup. Cross-reference data should map `{bookId: {chapterNumber: {verseNumber: [refBookId, refChapter, refVerse][]}}}`.
